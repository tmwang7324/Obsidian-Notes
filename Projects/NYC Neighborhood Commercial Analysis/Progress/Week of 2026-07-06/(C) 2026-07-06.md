# (C) 2026-07-06 — Fixed the deployed ranking error; learned Supabase config

## Done
- Diagnosed the **deployed ranking 500** (local worked, Railway/Vercel didn't): after migrating embeddings from OpenAI (1536-dim) to **Voyage `voyage-4`** (1024-dim), Railway was missing `VOYAGE_API_KEY`, so `resolve_embedding_backend()` silently fell back to the still-present `OPENAI_API_KEY` → embedded queries at 1536-dim → Postgres rejected the dimension mismatch against the `vector(1024)` `match_neighborhoods` RPC.
- **Fixed it** by adding `VOYAGE_API_KEY` on Railway and **pinning `EMBEDDING_BACKEND=voyage`** to prevent silent provider drift; verified via a `/api/rank` curl.
- Learned **Supabase** config for the first time: `config.toml` `project_id` is a local slug (not the remote ref); migrations are numbered `*.sql` files auto-run in filename order; secrets stay in env vars (`env(...)` substitution), never in `config.toml`; `link` writes to gitignored `.temp/`, `db push` applies only new migrations.
- Documented running the FastAPI backend locally (`uvicorn api.main:app --reload --port 8000`) and pointing the frontend at it via `NEXT_PUBLIC_API_URL`.

## Decisions
- **Decided:** Treat numbered `supabase/migrations/*.sql` as the single source of truth for schema; never hand-edit the hosted DB — [[(C) Migrations as Schema Source of Truth|Migrations as Schema Source of Truth]]
