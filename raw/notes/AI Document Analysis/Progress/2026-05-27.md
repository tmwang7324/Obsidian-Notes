
## Next.js: Context, Navigation, and Auth Sessions

---

### Exchange 1 — Can a Server Component use `useContext`?

**User:** Can server components consume React context with `useContext`?

**ChatGPT:** No. Reading context requires a [[Client vs Server Components | Client Component]] (`"use client"`). But *rendering* a context provider from a Server Component is fine — it does not make that server component a client component.

> Rule: **Render a provider from a Server Component → allowed. Read context values with `useContext` → Client Component required.**

For auth specifically, use context only for small UI changes (login button, avatar, username). For protected data, read and verify the session cookie directly in the Server Component:

```ts
// Server Component
const session = cookieStore.get("session")?.value;
const user = await adminAuth.verifySessionCookie(session);
```

### Exchange 2 — How do you use `next/navigation`?

**User:** how to use next/navigation

**ChatGPT:** `next/navigation` hooks are Client Component only. Key hooks:

| Hook | Use |
|---|---|
| `useRouter()` | Programmatic navigation after logic runs (e.g. post-login redirect) |
| `usePathname()` | Read the current path (e.g. highlight active nav link) |
| `useSearchParams()` | Read query params from the URL |
| `useParams()` | Read dynamic route segments (e.g. `[documentId]`) |

Prefer `<Link>` for clickable nav links; use `router.push()` only inside event handlers or after async logic.

Use `router.refresh()` after a mutation (e.g. file upload) to re-run Server Components and update data without a full page reload.

For the Firebase/session login flow specifically:
```ts
await createSession(idToken);   // sets HTTP-only cookie
router.replace("/dashboard");
router.refresh();
```

### Exchange 3 — Navigation tools for Server Components

**User:** are there any navigation tools for server components

**ChatGPT:** Server Components don't use hooks. Their navigation tools are:

| Tool | Use |
|---|---|
| `redirect("/path")` | Redirect during server render (equivalent of `router.push`) |
| `permanentRedirect("/path")` | Permanent route changes |
| `notFound()` | Render the 404 page when a resource doesn't exist |
| `<Link href="…" />` | Normal navigation links — works in Server Components |
| `params` / `searchParams` props | Read route segments and query params from page props |

Common pattern — protect a route server-side:
```ts
const session = cookieStore.get("session")?.value;
if (!session) redirect("/login");
```

### Exchange 4 — Detecting HTTP-only cookies on the frontend

**User:** I want to construct a Next.js + Firebase + Express app that handles session creation in the backend, but session verification both on the frontend and backend so there are no latency issues. How can I implement cookie detection in the frontend if these cookies are HTTP-only?

**ChatGPT:** You cannot read an HTTP-only cookie from frontend JavaScript — that is its entire security purpose. Instead, split auth into two layers:

| Layer | Mechanism | Where |
|---|---|---|
| Fast UI state | Firebase `onAuthStateChanged` | Next.js frontend |
| Authoritative session | `verifySessionCookie()` | Express backend |

The frontend detects the session by asking the backend — not by reading the cookie:
```ts
const res = await fetch("/api/me", { credentials: "include" });
// browser attaches the HTTP-only cookie automatically
// frontend only sees the response (user object or 401)
```

**Full flow:**
1. User signs in → Firebase returns an ID token
2. Frontend POSTs ID token to Express (`POST /auth/session-login`)
3. Express calls `admin.auth().createSessionCookie(idToken)` and sets it as HTTP-only
4. Frontend optimistically renders using Firebase client state
5. Every protected Express route runs `verifySessionCookie()` to authenticate

**Logout:** run `Promise.allSettled([signOut(auth), fetch("/auth/logout")])` in parallel — both sides clear independently.

### Exchange 5 — What is `getCookie` and what is a CSRF token?

**User:** *(Firebase docs snippet)* Where is the `getCookie` method imported and what is a `csrfToken`?

**ChatGPT:** `getCookie` is not from Firebase — it's a helper you define yourself:

```ts
function getCookie(name: string) {
  const cookie = document.cookie.split("; ").find(row => row.startsWith(`${name}=`));
  return cookie ? decodeURIComponent(cookie.split("=")[1]) : null;
}
```

A **CSRF token** prevents cross-site request forgery. Because browsers automatically attach cookies to requests, a malicious site could trick a logged-in user's browser into POSTing to your backend. A CSRF token proves the request came from *your* frontend.

Pattern (**double-submit cookie**):
1. Backend generates a `csrfToken` and sets it as a *readable* (non-HTTP-only) cookie
2. Frontend reads it via `getCookie("csrfToken")`
3. Frontend sends it in the request header (`CSRF-Token: <value>`)
4. Backend compares the header value against the cookie value — if they match, accept the request

Two cookies in play:

| Cookie | HTTP-only | Purpose |
|---|---|---|
| `session` | ✅ Yes | Authenticates the user — backend reads only |
| `csrfToken` | ❌ No | Proves request origin — frontend must read it |
